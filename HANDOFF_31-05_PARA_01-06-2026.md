## 5. Sobre DSPy / MIPROv2 / Prompt Caching

Você perguntou sobre essas três otimizações. Recomendação direta:

### 5.1 Prompt Caching (Claude SDK) — **fazer já, ganho imediato**

O `_MASTER_INSTRUCTION.md` cresceu pra ~620 linhas (seções 0 a 25). Toda
chamada da Lia hoje recompra o prompt inteiro a $3/Mtok input. Ligando o
`cache_control` do Anthropic SDK, esse bloco vira leitura quente (10% do
preço, hit em < 5min). Estimativa: redução de 40-70% do custo por
mensagem, sem mudar o comportamento.

**Como fazer**: no `voice_agent/responder.py`, na chamada `messages.create`,
marcar o system prompt com `cache_control: {"type": "ephemeral"}` no
último bloco do system. Subir env `ANTHROPIC_PROMPT_CACHING=on`.
Tempo: 1h. Risco: zero (degrada graciosamente se cache miss).

### 5.2 DSPy + MIPROv2 — **vale, mas só depois de coletar dataset**

DSPy compila prompts a partir de exemplos rotulados (entrada → saída
correta). MIPROv2 é o otimizador que faz busca bayesiana sobre instruções
e few-shots. Aplica-se bem aqui em problemas como:

- Classificar `motivo` em uma das 5 categorias (rotina / urgência / ...).
- Detectar pré-requisitos faltantes na conversa (data nasc / motivo / convênio).
- Gerar a frase de oferta de slot variando contexto.

**Pré-condição** que ainda não temos: dataset de **150-300 conversas
rotuladas** (entrada do paciente, saída ideal da Lia). Sem isso, MIPROv2
não tem o que otimizar.

**Caminho prático sugerido**:
1. Esta semana: ativar prompt caching (5.1) — ganho imediato.
2. Semana 2: instrumentar `responder.py` pra logar `(input, output, foi_correto?)` em SQLite local.
3. Semana 3: rotular 200 turnos (pode ser via Claude rodando em batch revisão).
4. Semana 4: compilar com DSPy/MIPROv2 a sub-rotina mais ruidosa (sugiro: classificação de motivo). Comparar prompt otimizado vs prompt mestre via A/B. Manter o que ganhar.

NÃO recomendo migrar o `responder.py` inteiro pra DSPy agora — perde a
auditabilidade do `_MASTER_INSTRUCTION.md` em código humano-legível. Use
DSPy só para módulos isolados (classificadores).

## 6.5 Otimização de prompt — INSTALADA (task #86)

`voice_agent/responder.py` agora separa o system em **2 blocos**:

1. **Bloco ESTÁVEL** (cacheado): `_base_system_prompt` — o
   `_MASTER_INSTRUCTION.md` (~30k tokens) com `cache_control:
   {"type":"ephemeral"}`. Anthropic cobra **10% do preço normal** em hits
   dentro de 5min. Economia esperada: **40-70% por mensagem** quando há
   tráfego contínuo (que é o caso da Lia).

2. **Bloco VARIÁVEL** (sem cache): `today_brt + caller_context +
   kb_block + RAG`. Esses mudam por chamada — não fazem sentido cachear.

Switches operacionais:

| Env | Default | Função |
|---|---|---|
| `ANTHROPIC_PROMPT_CACHING_DISABLED=1` | **off (cache ON)** | Kill switch — volta pra string concatenada (rollback rápido). |
| `MEMORIA_RAG_ENABLED=1` | **off (RAG NÃO injeta)** | Liga o RAG pra recuperar lições da memória ativa. Recomendo ativar amanhã, depois de observar 1-2 conversas reais. |

**Anti-sobrecarga garantida**:
- RAG limitado a 3 trechos × 800 chars = ~2.4k tokens extras max.
- Cutoff de similaridade 0.08 — se nada relevante, NÃO injeta nada.
- Import tardio do `memoria_rag` — scikit-learn só carrega quando RAG está on.
- Falha no RAG = log warning + segue sem memória (nunca quebra reply).

**Pytest**: 18 testes em `tests/test_responder_optimizations.py` cobrindo:
- Defaults do cache e do RAG.
- Estrutura do payload (lista vs string).
- Bloco variável aparece DEPOIS do estável (ordem do cache).
- RAG inject só quando habilitado.
- Falha do módulo RAG = fallback silencioso (não quebra reply).
